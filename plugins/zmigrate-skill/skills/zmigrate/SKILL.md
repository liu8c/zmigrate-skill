### 多知识库文件并存时的处理规则

后续 `references/` 目录中可能会加入多份知识库文件。遇到多文件时，按以下规则处理：

1. **先发现可用文档**
   - 优先识别成对出现的正文文件和索引文件：`*.md` + `*_INDEX.md`
   - 如果某份正文没有索引，也可以作为补充文档使用，但优先级低于带索引的文档

2. **优先级顺序**
   - 优先使用文件名中版本/日期更新的知识库（如 `20250410` 晚于 `20250301`）
   - 如果新版索引命中但正文缺失、乱码严重或内容不完整，再回退到较旧版本交叉验证
   - 如果多个版本都有命中，优先采用**更新版本**的结论，同时说明参考了哪些版本

3. **按主题补充**
   - 某个问题在主知识库里只命中索引或正文过短时，可继续查找其他知识库文件中的同主题条目
   - 如果一个文件命中主结论，另一个文件补充了步骤、日志或限制条件，应合并输出，但要说明来源文件

4. **冲突处理**
   - 如果不同版本结论冲突，优先报告最新版本
   - 同时提醒用户存在版本差异，并分别列出来源文件路径，避免把旧版本经验当作当前版本定论

## 工作流程

### 第一步：读取索引文件并识别可用文档

当用户询问时，先识别 `references/` 目录中有哪些知识库文件，再优先读取主索引文件快速定位内容：

```bash
# 先查看 references 目录中有哪些知识库文件
ls references/

# 优先读取最新主索引文件
Read references/ZMigrate知识库-20250410-压缩版_INDEX.md
```

### 第二步：优先按索引和页号定位正文

`ZMigrate知识库-20250410-压缩版.md` 是由 PDF 转换而来，存在按页拼接、断行、半角/全角混用和少量乱码的情况。**不要只依赖精确关键词 grep**，优先按索引命中的问题名，在主文档中结合页号标记 `## 第N页` 和上下文人工定位。

推荐做法：

1. 先在索引中确认问题名称存在。
2. 从主文档前几页目录区查找该问题名所在的目录页。
3. 顺着后续页号继续读取，直到在正文中找到对应标题或相邻上下文。
4. 若当前条目正文较短或被截断，继续读取下一页。
5. 若条目中引用了其他知识库条目（如“参考知识库《...》处理”），继续沿引用条目补读。

```bash
# 先读主文档前几页目录区，确认问题名是否出现在目录
Read references/ZMigrate知识库-20250410-压缩版.md

# 再按页号和上下文连续读取，而不是只做精确 grep
Read references/ZMigrate知识库-20250410-压缩版.md
```

### 第三步：关键词检索作为辅助手段

当正文标题未损坏或需要定位日志/参数名时，再使用搜索工具作为辅助，而不是唯一手段：

```bash
# 搜索同步问题详情
Grep "无法读取目标磁盘" references/ZMigrate知识库-20250410-压缩版.md

# 搜索切换问题详情
Grep "Server service is not ready" references/ZMigrate知识库-20250410-压缩版.md

# 搜索日志和参数说明
Grep "treker.log" references/ZMigrate知识库-20250410-压缩版.md
Grep "antenna.log" references/ZMigrate知识库-20250410-压缩版.md
Grep "等待目标机磁盘" references/ZMigrate知识库-20250410-压缩版.md
```

### 第四步：提供完整答案

结合索引、页号定位、辅助检索和多文件交叉验证结果，提供：
1. 问题的明确结论
2. 处理步骤或排查方法
3. 相关日志、参数或注意事项
4. 对应本地文档路径
5. 如有必要，标出关键证据的 `file_path:line_number`
6. 如果答案合并自多份知识库，明确区分主结论来源和补充来源

### 第五步：多文件交叉验证

当 `references/` 中存在多份知识库时，追加执行：

1. 若最新版本已经命中正文，仍可用旧版本补充步骤、限制条件或日志示例。
2. 若最新版本只命中索引、正文损坏或内容过短，继续读取旧版本同主题条目。
3. 输出时按“主结论来源 -> 补充来源”组织，避免混淆不同文档版本。
4. 如多个文档结论一致，可直接汇总结论；如存在冲突，要显式说明“不同版本知识库存在差异”。

## 常见问题快速参考

### 同步问题

| 场景 | 关键词 | 说明 |
|------|--------|------|
| 磁盘状态异常 | `无法读取目标磁盘<数字>状态` | 目标磁盘状态获取失败 |
| 任务卡住 | `读取工作已提交到来源服务器` | 同步流程卡顿 |
| 磁盘不可见 | `不显示源端主机的磁盘` | 源端磁盘识别异常 |
| 写入失败 | `无法将保护数据写入目标磁盘` | 目标盘写入或存储异常 |

### 切换问题

| 场景 | 关键词 | 说明 |
|------|--------|------|
| 服务未就绪 | `Server service is not ready` | 目标机服务未完成初始化 |
| 系统转换失败 | `Linux系统转换失败` | 目标机系统转换异常 |
| 磁盘脱机 | `Windows迁移后有磁盘脱机` | 目标机磁盘状态异常 |

### 排障日志

| 日志 | 位置 | 说明 |
|------|------|------|
| `antenna.log` | 源端主机诊断日志 | 分析源端读取、快照、采集问题 |
| `treker.log` | 目标侧或数据网关诊断日志 | 分析目标机创建、磁盘状态、写入和回报问题 |
| 同步任务诊断日志 | 管理控制台导出 | 分析任务整体执行链路 |

## 响应策略

1. **优先使用索引文件**: 索引文件提供分类清晰的问题列表，快速定位相关内容
2. **页号/上下文优先于精确匹配**: 对 PDF 转 md 的正文，优先按 `## 第N页` 和相邻上下文定位，再用 grep 辅助
3. **提供完整结论**: 包含结论、处理步骤、证据片段和本地文件路径
4. **版本注意**: 如果后续引入多个版本文档，要说明版本差异和适用范围
5. **日志结合分析**: 同步和切换问题优先结合 `antenna.log`、`treker.log` 和任务诊断日志判断
6. **多文档合并要标来源**: 如果答案来自多份知识库，明确区分主结论和补充来源

## 注意事项

1. **优先使用 md 文档**: references 目录中的知识库文档统一按 `*.md` 组织
2. **先索引后正文**: 先读 `ZMigrate知识库-20250410-压缩版_INDEX.md`，再去主文档定位细节
3. **正文优先按页读**: 主文档是 PDF 转 md，常见按页拼接、断行、字符变形和少量乱码，正文定位优先看 `## 第N页` 与上下文，不要只依赖精确 grep
4. **相邻条目要继续追踪**: 如果条目中出现“参考知识库《...》处理”，继续读取被引用条目，补全结论
5. **版本兼容**: 注意不同版本知识库中的结论可能存在差异
6. **证据来源**: 输出时要保留本地文件路径，便于复核；命中正文时优先给出 `file_path:line_number`
7. **未命中不要硬答**: 如果只命中索引、没有命中正文，要明确说明是“索引命中但正文待确认”，不要把推测当成知识库原文结论
